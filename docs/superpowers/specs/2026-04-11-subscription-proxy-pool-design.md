# 订阅转固定 HTTP/SOCKS5 代理池系统设计

日期：2026-04-11  
状态：已确认设计，待实现计划  
目标版本：v1

## 1. 目标

构建一个部署在远程 VPS 上、面向单用户自用的代理池服务。系统消费机场/节点订阅，自动解析与维护上游节点，对外长期提供固定的 HTTP 与 SOCKS5 代理入口，并通过管理员后台管理订阅、节点、切换与运行状态。

v1 的核心目标：

- 输入为机场/节点订阅链接
- 部署方式以 Docker 为主
- 对外同时提供 HTTP 与 SOCKS5 固定入口
- 单入口背后自动维护和切换上游节点
- 支持 Clash 与 V2Ray 常见订阅/节点格式
- 以稳定性优先，尽量避免长期使用无效节点
- 管理后台支持基础热生效；不适合热更新的配置明确提示需重启服务
- 面向公网 VPS 使用，带管理员认证和代理入口认证

## 2. 非目标

v1 不包含以下内容：

- 多用户系统
- 多租户隔离
- 多订阅优先级编排
- 自研代理协议内核
- 复杂规则路由
- 流量计费/配额系统
- 集群化或高可用主备部署
- 访客模式或角色权限系统
- 高级拓扑可视化

## 3. 总体架构

系统采用控制面与数据面分离架构。

### 3.1 控制面（Go 服务）

负责：

- 拉取订阅
- 解析 Clash / V2Ray 常见格式并归一化
- 持久化节点、检查结果、当前活动节点与系统配置
- 定时健康检查与节点评分
- 节点分层与活动节点选择
- 生成 sing-box 配置并触发 reload
- 提供管理员 Web UI、认证、日志与管理操作

### 3.2 数据面（sing-box）

负责：

- 承载上游代理节点
- 对外暴露固定 HTTP 与 SOCKS5 入口
- 根据控制面下发的配置使用当前活动节点作为主要出站
- 在控制面触发时进行 reload，尽量不中断服务

### 3.3 存储层（SQLite）

负责持久化：

- 订阅信息
- 节点注册表
- 检查结果与运行状态
- 当前活动节点
- 可配置参数
- 操作日志与切换事件

### 3.4 对外接口

v1 对外暴露三个接口：

- 固定 HTTP 代理入口
- 固定 SOCKS5 代理入口
- 管理员 Web UI

## 4. 技术选型

### 4.1 选型

- 主语言：Go
- 数据面内核：sing-box
- 数据库：SQLite
- 部署方式：Docker / Docker Compose
- Web UI：Go 服务内嵌静态资源的轻量前端

### 4.2 选型理由

- Go 适合长期运行的后台服务、并发检查和 Docker 部署
- sing-box 适合作为稳定的数据面，避免自研协议处理
- SQLite 满足单用户单实例场景，维护成本低
- 内嵌式 Web UI 可降低前后端分离复杂度，便于 v1 落地

## 5. 核心模块划分

### 5.1 subscription

负责订阅管理：

- 保存订阅 URL
- 定时拉取订阅内容
- 记录最后拉取结果
- 触发后续解析流程

### 5.2 parser

负责解析并归一化节点：

- 识别 Clash 订阅格式
- 识别 V2Ray/分享链接等常见格式
- 转换为内部统一节点模型

### 5.3 registry

节点注册表：

- 节点去重
- 节点启用/禁用管理
- 记录来源、元信息、状态、层级
- 维护节点生命周期字段

### 5.4 checker

健康检查器：

- 执行定时节点检测
- 更新延迟、成功率、最近状态
- 对活动节点执行更高优先级检查

### 5.5 scorer

评分与分层：

- 计算综合评分
- 根据得分将节点分入 L1/L2/L3
- 为选择器提供排序结果

### 5.6 selector

活动节点选择器：

- 自动选出当前活动节点
- 当前节点异常时同层切换
- 同层无可用节点时跨层降级
- 在策略允许时执行回切

### 5.7 runtime

运行时编排：

- 根据当前活动节点生成 sing-box 配置
- 调用 sing-box reload
- 跟踪 reload 成功/失败结果
- 维护“当前运行状态”和“待重启配置”标记

### 5.8 webui/api

管理员后台：

- 登录认证
- 订阅管理
- 节点列表与详情
- 手动切换与启停控制
- 配置修改
- 事件与日志查看

## 7. 数据流设计

### 7.1 启动流程

1. Go 服务启动
2. 加载配置与 SQLite 数据
3. 恢复订阅、节点和当前活动节点状态
4. 启动 sing-box
5. 启动定时任务：订阅更新、健康检查、状态清理
6. 启动管理员 Web UI

### 7.2 订阅更新流程

1. 定时拉取订阅内容
2. 解析为统一节点模型
3. 与现有节点做差异比较
4. 新节点入库
5. 已消失节点标记为 removed
6. 仍存在节点更新元数据
7. 将新增/变化节点加入待检测队列
8. 重新评分与分层
9. 如触发切换条件，则进入选择与 reload 流程

### 7.3 健康检查流程

检查优先级：

1. 当前活动节点
2. 最近失败节点
3. 其余节点按轮询策略检查

检查结果写回：

- latency_ms
- success/failure
- recent_success_rate
- consecutive_failures
- last_success_at / last_failure_at
- cooldown_until

### 7.4 节点切换流程

触发条件：

- 活动节点健康检查失败
- 活动节点连续失败达到阈值
- 活动节点从订阅中消失
- 管理员手动切换
- 存在显著更优节点且满足保守切换条件

步骤：

1. selector 选择候选节点
2. runtime 生成新 sing-box 配置
3. 对配置做预校验
4. 触发 sing-box reload
5. reload 成功后更新当前活动节点
6. 写入切换事件日志

## 8. 分层、评分与切换策略

### 8.1 分层模型

v1 使用三级分层：

- L1：优选层
- L2：备援层
- L3：冷备层

### 8.2 节点状态

每个节点维护以下状态：

- active：当前可参与选择
- degraded：可用但质量下降
- cooldown：刚失败，暂时不参与选择
- disabled：长期失败、手动禁用或已移除

### 8.3 评分维度

v1 使用综合评分，不只依赖单一延迟：

- latency
- recent_success_rate
- consecutive_failures
- last_success_at
- check_recency

评分原则：

低延迟、高成功率、最近稳定、连续失败少的节点得分更高。

### 8.4 故障切换

故障切换应激进：

- 优先在当前层内选择下一个最高分节点
- 当前层无可用节点时降级到下一层
- 所有层都不可用时进入告警状态

### 8.5 优化切换

优化切换应保守，仅在以下条件同时满足时执行：

- 新节点评分明显优于当前活动节点
- 当前活动节点已运行超过最小持有时间
- 最近没有频繁切换

### 8.6 回切策略

默认不在高层节点恢复后立即抢占。仅当其优势明显且满足最小切换间隔时回切，以避免抖动。

### 8.7 手动覆盖

管理员可：

- 指定某节点为当前活动节点
- 临时禁用某节点
- 强制重新评分
- 解除手动锁定，恢复自动模式

## 9. 热更新与重启边界

v1 对热更新不做绝对承诺，而是区分可热生效和需重启生效。

### 9.1 尽量热生效

- 订阅更新
- 节点切换
- 健康检查周期
- 评分阈值
- 自动回切开关
- 手动启用/禁用节点

### 9.2 可能需重启生效

- 监听地址变化
- 监听端口变化
- 某些 sing-box 基础入站结构变更
- 部分影响进程启动参数的认证或基础运行项

后台需要明确显示：

- 已热生效
- 已保存，重启服务后生效

## 11. 安全设计

部署目标是远程 VPS，默认按“公网可达”场景设计。

### 11.1 管理后台认证

- 单管理员账号密码
- 密码仅存哈希
- 会话应有过期时间
- 首次使用时强提醒修改弱密码

### 11.2 代理入口认证

- HTTP 与 SOCKS5 入口支持认证
- 默认建议开启
- 后台明确提示未开启认证时的风险

### 11.3 审计与敏感信息处理

- 关键操作写入审计日志
- 前端不直接暴露完整订阅原文
- 敏感认证字段不明文回显

## 12. 错误处理

### 12.1 订阅拉取失败

- 不覆盖当前可用节点集
- 保留上次成功结果继续服务
- 标记订阅异常
- 记录事件日志
- 下个周期自动重试

### 12.2 健康检查失败

- 增加失败计数
- 超过阈值则进入 cooldown 或 disabled
- 如为活动节点则立即触发切换

### 12.3 sing-box reload 失败

- 保留旧配置继续运行
- 不更新当前活动节点状态
- 记录失败原因和时间
- 后台展示错误信息

### 12.4 数据库异常

- 启动期异常直接失败退出
- 运行时写入异常必须记录
- 不得把控制面持久化失败误判为切换成功

### 12.5 配置错误

- 后台保存前先校验
- reload 前生成临时配置并做预校验
- 对不支持热生效的变更标记 restart_required

- HTTP 入口可用

### 13.4 长稳验证

重点验证：

- 长时间运行时不会卡死
- 不会频繁抖动切换

## 14. v1 范围总结

### 14.1 包含

- 单用户
- 单主订阅
- Docker 部署
- HTTP + SOCKS5 固定入口
- 管理员 Web UI
- 自动订阅更新
- 自动健康检查
- 节点评分、分层、自动切换
- 基础认证
- 尽量热更新，必要时提示重启

### 14.2 不包含

- 多用户
- 多订阅编排
- 复杂路由规则
- 多地区高级策略

- 高级权限系统
- 集群高可用
- 自研协议内核

## 15. 实施原则

v1 的设计原则如下：

- 稳定优先于理论最优
- 故障切换激进，性能优化切换保守
- 控制面与数据面边界清晰
- 对外始终暴露固定入口，内部吸收节点波动
- 能热更新的热更新，不能热更新的明确提示重启

## 16. 一句话定义

这是一个部署在远程 VPS 上的单用户订阅代理池服务：消费机场订阅，自动筛选和维护可用节点，对外长期提供稳定的 HTTP / SOCKS5 固定代理入口，并通过管理员后台进行管理。
